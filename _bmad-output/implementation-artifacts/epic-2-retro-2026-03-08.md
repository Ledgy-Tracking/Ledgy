## 🔍 Cross-Story Patterns

### Common Strengths

1. **Multi-round code review works** — Every reviewed story caught HIGH-priority bugs. The pattern of implementing → reviewing → fixing → re-reviewing consistently produced better outcomes.
2. **Reusable component extraction** — Story 2-2 created `EmptyState` and `LoadingSkeleton` as genuinely shared components. Story 2-7/2-8 shared the `isTauri()` detection pattern and `Function()` constructor approach.
3. **"What's already done vs. what's your job" dev notes** — Stories 2-7 and 2-8 explicitly called out existing implementations to protect. This prevented 3 accidental re-implementations.
4. **Accessibility first** — WCAG 2.1 AA implemented in 2-2 and 2-5 without being an afterthought. Multiple ARIA and keyboard fixes surfaced in reviews.

### Recurring Challenges

1. **Stateful reset bugs in confirmation flows** — Story 2-5 needed 3 rounds to get all state reset paths right (deleteConfirmName, showForceLocal, purgeRemote all needed explicit resets on both cancel AND success paths). This is a predictable failure mode for multi-state dialogs.
2. **Story file / sprint-status synchronization failures** — Three stories (2-2, 2-3, 2-6) had sprint-status updated without corresponding story file updates. The commit discipline required by `project-context.md` was not followed consistently.
3. **Story ordering risks** — Story 2-4 was implemented with a dependency on Story 2-3's route (`/profiles/new`) before Story 2-3 was formally complete. This created a hidden coupling that could mask incomplete work.
4. **Carried-over Epic 1 commitments ignored** — Integration tests and E2E tests were committed as action items from Epic 1 for implementation in Epic 2. Neither was added in any Epic 2 story.

## ✅ Retrospective Closure

### Key Takeaways

1. **Multi-round code review is non-negotiable for safety-critical and stateful UI** — Story 2-5's 3-round journey caught bugs that would have caused real user data loss (stale state after deletion). The cost of extra review rounds is far lower than the cost of shipping broken confirmation dialogs.
2. **Sprint-status hygiene directly risks feature completeness** — Marking stories `done` in sprint-status without updating story files created 3 unknown-state situations heading into Epic 3. The process must be reversed: story files first, sprint-status last.
3. **Carried-over commitments don't auto-complete** — Integration tests and E2E tests were committed in Epic 1 and carried into Epic 2 untouched. In Epic 3, these must be story-level tasks, not aspirational footnotes.
4. **Epic 3 is the highest-complexity epic yet** — 16 stories across schema builder, Data Lab virtualization, bidirectional relations, and undo/redo. The team must enter it with solid profile switching (2-6), a verified creation flow (2-3), and at least minimal integration test infrastructure.
